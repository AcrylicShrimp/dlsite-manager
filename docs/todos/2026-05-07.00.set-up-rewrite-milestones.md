## Context
The rewrite needs staged milestones before implementation: a Rust DLsite API v3 crate, a download and decompression handling crate or crate pair, a SQLite-backed storage crate, and the final Tauri GUI adapter. Legacy Rust code in ../dlsite-manager contains download, extraction, scanning, and storage behavior. The newer Flutter reference in ../dlsite-manager.rework-v3-riif contains the clearer DLsite API v3 login and product-listing flow but lacks download/storage/full workflows. Open design question: download and decompression may be one crate if decompression is an internal post-download concern, or separate crates if archive handling should be reusable and testable independently.

## Why It Is Needed
Milestones are needed to keep the rewrite from recreating the old maintenance-heavy structure. Each core area must be implemented as a reusable Rust crate with automated tests, using environment-injected secrets and pinned product IDs for live DLsite verification where needed. The Tauri layer should remain a thin adapter over these crates.

## Completion Criteria
- [ ] API v3 crate exposes login/session, product count, purchased works, work details, and discovered download API surfaces.
- [x] API tests are automated and use .env.example-documented environment variables for credentials and pinned product IDs.
- [ ] Download/decompression crate boundary is decided and documented.
- [ ] Download/decompression behavior is covered by automated tests with env-configured fixtures or pinned DLsite works.
- [ ] Storage crate uses SQLite and manages works, purchases, settings, accounts/session data, and download status.
- [ ] Storage tests cover schema creation, CRUD/query behavior, and migration expectations.
- [ ] GUI milestone uses API, download/decompression, and storage crates through a thin Tauri adapter.

## Detailed Plan
1. Create workspace structure for reusable Rust crates before porting behavior.
2. Implement dlsite API v3 crate first, using the Flutter dm_api package as the primary reference for login, content/count, content/sales, and content/works.
3. Investigate and add missing v3 download APIs with live tests gated by env vars.
4. Decide whether decompression belongs in the download crate or in an archive/decompression crate; prefer separation if archive behavior is reusable outside DLsite downloads.
5. Implement download/decompression tests using small deterministic local fixtures plus optional env-gated live tests.

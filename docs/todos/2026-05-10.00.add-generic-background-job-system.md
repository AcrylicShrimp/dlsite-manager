## Context
The current GUI can trigger account sync, but `sync_account` is still a blocking Tauri command. `dm-library` already exposes sync progress and cooperative cancellation primitives, and `dm-download` has its own progress/cancellation model. If the app keeps adding one-off command/event pairs for sync, download, extraction, scanning, metadata refresh, and future library tasks, the Tauri layer will become the coordination layer again.

The intended architecture requires Tauri to stay thin. A generic background job system should own lifecycle mechanics such as dispatch, active/finished tracking, cancellation requests, bounded logs, event publication, and UI snapshots. Domain crates should still own the real work: account sync in `dm-library`, downloads in `dm-download`, archive extraction in `dm-archive`, and storage writes in `dm-storage`.

The first implementation should focus on replacing account sync with a job-backed flow. Download and archive jobs should be designed for, but wired in only after the account-sync path proves the job model.


## Why It Is Needed
The app needs long-running operations that remain observable and cancellable from the UI. Account sync can involve login, count lookup, purchase listing, batched work details, and cache commits. Downloads and archive extraction can be much longer and need progress, cancellation, logging, and eventual persistent download-state integration.

A shared job model prevents every workflow from inventing its own Tauri command shape, progress event name, cancellation path, and status display. It also gives the frontend one place to show active work and recent failures while keeping network, filesystem, archive, and storage behavior inside reusable Rust crates.


## Completion Criteria
- [ ] A design document under `docs/designs/` defines the background job lifecycle, event schema, cancellation semantics, logging model, Tauri command boundary, and persistence policy.
- [ ] A reusable `dm-jobs` crate exists with automated tests for job start, success, failure, cooperative cancellation, snapshot listing, per-job log ordering, event ordering, and clearing finished jobs.
- [ ] `dm-jobs` does not contain DLsite, storage, download, archive, or frontend-specific behavior.
- [ ] The Tauri app owns a `JobManager` in app state and exposes generic job commands such as start/list/get/cancel/logs/clear.
- [ ] Account sync is started through the job system instead of a blocking `sync_account` command.
- [ ] Account-sync progress from `dm-library` is translated into generic job progress/events without moving sync logic into Tauri.
- [ ] The frontend subscribes to one generic job event stream and renders active/recent jobs with progress, status, errors, logs or log summaries, and cancel actions.
- [ ] Cancellation requests from the UI reach `dm-library` and are recorded as cancelled sync runs when cancellation is observed.
- [ ] Finished account-sync jobs refresh affected account/product cache views in the frontend.
- [ ] Validation covers Rust unit tests, workspace check/tests, frontend typecheck/build, and a Tauri dev smoke run.
- [ ] Documentation and work log are updated, and the checkpoint is committed.


## Detailed Plan
1. Finalize the architecture document.
   - Create `docs/designs/YYYY-MM-DD.NN-background-job-system.md`.
   - Define the core terms: job, job kind, status, phase, progress, event, log entry, snapshot, cancellation token, and result.
   - Decide whether `dm-jobs` is purely in-memory for the first cut. Recommended first cut: in-memory job history only; durable domain facts remain in tables such as `sync_runs` and future download-status tables.
   - State that normal "abort" means cooperative cancellation. Do not hard-kill tasks during normal operation because sync/download/extraction need cleanup and transactional boundaries.

2. Add the `dm-jobs` crate.
   - Types:
     - `JobId`;
     - `JobKind`;
     - `JobStatus` with `Queued`, `Running`, `Cancelling`, `Succeeded`, `Failed`, and `Cancelled`;
     - `JobProgress` with optional current/total/unit;
     - `JobSnapshot`;
     - `JobEvent`;
     - `JobLogEntry`;
     - `JobError`;
     - shared cooperative `CancellationToken`.
   - Manager API:
     - `spawn(kind, title, metadata, async job_fn) -> JobId`;
     - `list_jobs() -> Vec<JobSnapshot>`;
     - `get_job(id) -> Option<JobSnapshot>`;
     - `cancel_job(id) -> CancelResult`;
     - `job_logs(id, after_seq, limit) -> JobLogPage`;
     - `clear_finished()`.
   - Runtime behavior:
     - assign monotonically increasing event/log sequence numbers;
     - retain active jobs plus a bounded finished-job history;
     - retain bounded per-job logs;
     - expose an event receiver/broadcast mechanism for Tauri;
     - update snapshots atomically around status/progress/log changes.
   - Tests:
     - successful job transitions to `Succeeded`;
     - failing job transitions to `Failed` with structured error text/code;
     - cancellation request transitions through `Cancelling` and ends as `Cancelled` when observed;
     - logs are ordered and bounded;
     - events are ordered and include final snapshots;
     - clearing finished jobs does not affect running jobs.

3. Unify cancellation primitives carefully.
   - Avoid duplicating cancellation token implementations across crates.
   - Prefer one shared token type that domain crates can accept without depending on Tauri.
   - If depending on `dm-jobs` from domain crates feels too broad during design, split the token/progress primitives into a smaller crate before implementation. Do not let Tauri define the cancellation type.

4. Wire Tauri to the job manager.
   - Add `JobManager` to `AppState`.
   - Register commands:
     - `start_account_sync(request) -> StartJobResponse`;
     - `list_jobs() -> Vec<JobSnapshotDto>`;
     - `get_job(job_id) -> JobSnapshotDto`;
     - `cancel_job(job_id) -> JobSnapshotDto`;
     - `get_job_logs(job_id, after_seq, limit) -> JobLogPageDto`;
     - `clear_finished_jobs()`.
   - Emit one generic event, likely `dm-job-event`, instead of workflow-specific event names.
   - Keep existing account/product commands for data access.
   - Remove or deprecate the blocking `sync_account` command once the frontend no longer uses it.

5. Adapt account sync into a job.
   - `start_account_sync` validates account ID and optional one-shot password, then spawns a job.
   - The job creates a `DlsiteClient`, wraps it in `DlsiteSyncSource`, and calls `dm-library`.
   - Map `dm-library::SyncProgress` into generic job phases:
     - `loggingIn`;
     - `loadingCount`;
     - `loadingPurchases`;
     - `loadingWorks`;
     - `committing`;
     - `completed`.
   - Add job logs for high-value milestones and failures without logging secrets, cookies, serial numbers, or raw passwords.
   - On completion, include account ID, sync run ID, purchased count, cached work count, page limit, and concurrency in job output metadata.

6. Update the frontend job model.
   - Add a small job store that subscribes to `dm-job-event`.
   - On app startup, call `list_jobs()` so the UI recovers current in-memory state after reload.
   - Replace direct `sync_account` calls with `start_account_sync`.
   - Show active/recent jobs in the Products screen, likely as a compact activity panel or account-row status.
   - Add cancel buttons for cancellable running jobs.
   - Refresh accounts/products after relevant successful sync jobs.
   - Show job failure messages and recent log lines without making logs the primary UI.

7. Add observability rules.
   - Use structured job events for UI state.
   - Use bounded per-job logs for user-visible diagnostics.
   - Use `tracing` only if the repo already has or explicitly adopts an app-wide tracing setup; do not introduce broad telemetry dependencies in the first cut unless needed.
   - Ensure secret-bearing values are never written to job logs or event metadata.

8. Defer download/archive integration until the sync path is stable.
   - After account sync proves the model, add `start_work_download`.
   - Map `dm-download::DownloadProgress` to the same generic progress shape.
   - Throttle/coalesce byte progress events before emitting to the frontend.
   - Connect future download-status persistence in `dm-storage`; do not treat generic job history as durable download truth.

9. Validate and checkpoint.
   - Run `cargo fmt --all`.
   - Run targeted `dm-jobs` tests.
   - Run `cargo test -p dm-library -p dm-storage`.
   - Run `cargo check --workspace`.
   - Run `pnpm check` and `pnpm build`.
   - Run `pnpm tauri dev` for a smoke test if the GUI command wiring changed.
   - Update `docs/work-log.md` and relevant design/todo docs.
   - Commit the checkpoint.

## Context
The current Svelte frontend is concentrated in src/routes/+page.svelte, which is over 6,000 lines and mixes DTO types, Tauri command calls, formatting helpers, view state, page markup, dialogs, and CSS. The user wants reusable component sets and dedicated page components before further GUI growth.

## Why It Is Needed
The all-in-one page makes UI patterns hard to reuse consistently. Recent Library-specific layout/background fixes show that shared page templates and common components are missing.

## Completion Criteria
- [ ] Shared frontend types, constants, and pure helper functions are moved out of `+page.svelte`.
- [ ] Reusable shell/overlay components exist for app layout, navigation, workspace header, toasts, and dialogs.
- [ ] Library UI is split into dedicated components without behavior changes.
- [ ] Downloads, Accounts, Activity, and Settings views are split into dedicated components without behavior changes.
- [ ] Tauri command wrappers and state orchestration are separated only after UI extraction proves the shape.
- [ ] Each slice passes `pnpm check`, `pnpm build`, and `git diff --check`.
- [ ] Refactor commits remain behavior-preserving and reviewable.

### Phase 1: Pure Type And Helper Extraction

Move behavior-free TypeScript pieces first:

- DTO and view types into `src/lib/model/types.ts`.
- filter constants and DLsite product-type metadata into `src/lib/model/constants.ts`.
- date, bytes, nullable string, app-info, and error formatting helpers into `src/lib/utils/format.ts`.
- product type, age label/tone, credit grouping, and detail helper functions into `src/lib/utils/products.ts`.
- job labels, queue summaries, progress percentages, and job output parsing into `src/lib/utils/jobs.ts`.
- account label/status/count helpers into `src/lib/utils/accounts.ts`.

Keep all Svelte state, command calls, event listeners, and markup in `+page.svelte` during this phase.

### Phase 2: Shared Shell And Overlay Components

Extract reusable UI that has clear boundaries and low coupling:

- `SidebarNav.svelte` for view navigation.
- `WorkspaceHeader.svelte` for page title/eyebrow.
- `ToastStack.svelte` for toast rendering/dismissal.
- `ConfirmationDialog.svelte` for generic confirmation UI.
- `BulkDownloadDialog.svelte` for bulk preview/notice UI.

`+page.svelte` should still own the dialog/toast state and pass data plus callbacks into these components.

### Phase 3: Library Component Split

Split the Library view without changing behavior:

- `LibraryView.svelte` receives products, filter state, jobs, facets, and callbacks.
- `LibraryControls.svelte` renders the sticky search/action shell.
- `LibraryFilters.svelte` renders expanded filters as normal page content.
- `ProductCard.svelte` renders one product row/card.
- `ProductActionMenu.svelte`, `ProductDetailDialog.svelte`, and `ProductImagePreview.svelte` render existing overlays.

Keep controller logic in `+page.svelte` until component boundaries are stable.

### Phase 4: Secondary Page Components

Extract the remaining pages:

- `DownloadsView.svelte` and `DownloadQueueRow.svelte`.
- `AccountsView.svelte`, `AccountSourceRow.svelte`, and `AccountEditor.svelte`.
- `ActivityView.svelte`, `JobList.svelte`, and `AuditLogList.svelte`.
- `SettingsView.svelte`.

Each page component should receive plain props and callbacks. Avoid adding global stores in this phase.

### Phase 5: Command And State Orchestration Cleanup

Only after UI extraction is stable:

- Add typed Tauri wrappers in `src/lib/api/tauri.ts`.
- Consider small controller modules for Library, Accounts, Jobs, Audit, and Settings.
- Keep command wrappers thin and avoid hiding long-running job behavior behind opaque abstractions.

### Validation And Commit Policy

- Run `pnpm check`, `pnpm build`, and `git diff --check` after every slice.
- Use `cargo check -p dlsite-manager` only when Tauri command DTOs or Rust-facing contracts change.
- Commit each phase or sub-phase separately.
- Do not mix visual redesign or new functionality into refactor commits.

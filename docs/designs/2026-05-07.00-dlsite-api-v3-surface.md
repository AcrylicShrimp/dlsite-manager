# DLsite API v3 Surface

## Design Goal

Define the first reusable Rust crate for the rewrite: a v3-first DLsite API crate that can be tested independently from Tauri, storage, filesystem downloads, and archive extraction.

The crate should expose authenticated DLsite API operations, session import/export, observed download resolution behavior, and authenticated byte streams. It should not write files or persist account/session data.

## Agreed Architecture

Create a dedicated Rust crate, tentatively `dm-api`, with two internal layers:

1. Raw discovery layer.
   - Sends exact DLsite v3 requests.
   - Captures status, headers, redirect `Location`, content type, and small diagnostic body snippets where useful.
   - Exists for tests and endpoint investigation.
   - Does not become the app-facing contract.

2. Stable typed layer.
   - Exposes login/session operations.
   - Exposes known v3 content operations.
   - Exposes conservative download resolution classification.
   - Exposes authenticated byte streams without filesystem ownership.

Known v3 content APIs from the Flutter reference:

- `GET https://play.dlsite.com/api/v3/content/count?last=N`
- `GET https://play.dlsite.com/api/v3/content/sales?last=N`
- `POST https://play.dlsite.com/api/v3/content/works`

Observed `content/count` shape:

```json
{"user":1223,"production":0,"page_limit":50,"concurrency":500}
```

`page_limit` is the current authoritative batch-size hint for `content/works`. `concurrency` is exposed by the API crate but should not be treated as a download-job policy until its exact meaning is verified.

Observed or hypothesized download entry point requiring verification:

- `GET https://play.dlsite.com/api/v3/download?workno=WORK_ID`

Initial download behavior should be classified from redirect targets instead of overfitting a premature manifest schema:

- Direct archive redirect to `https://www.dlsite.com/home/download...`
- Serial-required redirect to `https://www.dlsite.com/home/serial...`
- Split archive redirect to `https://www.dlsite.com/home/split...`
- Split archive redirect to `https://www.dlsite.com/home/download/split...`
- Serial-required redirect to `https://www.dlsite.com/home/download/serial...`
- Unknown redirect location
- Unavailable/not-authorized/error response

## Public Surface

Initial public API shape should be close to:

```rust
client.login(credentials).await;
client.export_session().await;
client.import_session(session).await;
client.validate_session().await;

client.content_count(ContentQuery { last }).await;
client.sales(ContentQuery { last }).await;
client.works(ids).await;
client.works_batch(ids).await;

client.resolve_download(work_id).await;
client.open_download_stream(download_ref, range).await;
```

`works(ids)` should auto-chunk requests. `works_batch(ids)` should send exactly one request and exist mainly for verification and lower-level callers.

Download resolution should initially use a conservative enum:

```rust
enum DownloadResolution {
    Direct { stream_request: DownloadStreamRequest },
    Split { location: Url },
    SerialRequired { location: Url },
    UnknownRedirect { location: Url },
    Unavailable { reason: DownloadUnavailableReason },
}
```

Split downloads should remain `Split { location }` until tests prove whether there is a v3 JSON backing endpoint or whether the split page requires HTML parsing.

## Invariants And Boundaries

- `dm-api` must not depend on Tauri.
- `dm-api` must not depend on the storage crate.
- `dm-api` must not write files.
- `dm-api` must not decompress archives.
- `dm-api` owns HTTP auth/session mechanics and authenticated byte streams.
- The future download crate owns retries, paths, temp files, resume policy, integrity, progress, cancellation, and calls into decompression.
- Session/cookie state must be exportable/importable so the future storage crate can persist it.
- Live tests must be skipped by default and enabled explicitly with an environment variable.
- Secrets and pinned product IDs must come from env vars documented in a crate-local `.env.example`.
- Typed contracts should harden only after raw tests prove DLsite behavior.

## Batch Size Detection

The expected current works batch size is 50, but this should not become a permanent invariant.

`works(ids)` should:

- Cache a detected batch limit in the client.
- Prefer `content/count.page_limit` when available.
- Keep passive detection from response headers or error bodies as a fallback.
- Fall back to a conservative known default only when detection is unavailable.
- Retry with a smaller detected limit if DLsite returns a parseable batch-limit error.

Tests should cover batch chunking without live credentials, and optional live tests should verify the detected limit is usable.

## First Implementation Observations

- The initial `dm-api` crate uses workspace resolver `3`.
- Live `content/works` responses can contain `regist_date: null`; typed work registration dates must therefore be optional.
- Live `content/works` responses can contain `genre_ids: null` and `tags: null`; typed DTOs currently normalize these to empty vectors.
- Live `content/count` responses include `production`, `page_limit`, and `concurrency`; `dm-api` caches `page_limit` as the works batch limit during session validation and explicit count calls.
- Live `content/works` with a large pinned sales-derived ID list can return fewer work details than requested while still succeeding. Callers should treat returned works as a subset of requested IDs and decide separately how to represent missing detail records.
- The tested login/session flow is based on the Flutter v3 reference and verifies the session by calling the v3 content count endpoint.
- Direct download stream opening may need to follow additional 302 redirects after the initial v3 download redirect.
- Download handling remains conservative: the API crate classifies direct, split, and serial-required redirects and exposes authenticated direct streams, while split archive structure and serial-required follow-up behavior remain future investigation items.

## Test Strategy

Always-run tests:

- DTO decoding from checked-in fixtures.
- Session snapshot serialization/deserialization.
- Redirect classification from fixture responses.
- Batch chunking and batch-limit detection behavior.
- Error parsing for unauthorized, unavailable, and unexpected response shapes.

Live tests:

- Skipped unless `DMSITE_API_TEST_LIVE=1`.
- Use env-injected credentials and pinned work IDs.
- Verify login and session reuse.
- Verify content count with and without `last`.
- Verify sales with and without `last`.
- Verify works auto-chunking.
- Verify download redirect classification.
- Verify direct archive stream access with a small ranged read.

Suggested crate-local `.env.example`:

```env
DMSITE_API_TEST_LIVE=0
DMSITE_API_TEST_USERNAME=
DMSITE_API_TEST_PASSWORD=
DMSITE_API_TEST_OWNED_WORK_ID=
DMSITE_API_TEST_DIRECT_DOWNLOAD_WORK_ID=
DMSITE_API_TEST_SPLIT_DOWNLOAD_WORK_ID=
DMSITE_API_TEST_SERIAL_REQUIRED_WORK_ID=
DMSITE_API_TEST_BATCH_WORK_IDS=
```

Split and serial-required live tests should be optional because not every test account will have those cases.

## Tradeoffs And Rejected Alternatives

- Rejected a v2/legacy compatibility API surface. Previous endpoints are assumed closed until proven otherwise, and the rewrite should find the v3 equivalents instead of porting old endpoint contracts.
- Rejected placing download filesystem behavior in `dm-api`. Authenticated byte streams belong in `dm-api`; file writing and job policy belong in the future download crate.
- Rejected finalizing a rich split-download manifest before observing actual v3 behavior.
- Accepted a raw discovery layer to keep investigation testable without committing unstable response shapes to the application contract.

## Remaining Open Issues

- Exact response and redirect shapes for `GET /api/v3/download?workno=...` need live verification.
- Split download structure after the redirect page is unknown.
- Serial-required behavior after the redirect page is unknown.
- `content/count.page_limit` is the observed dynamic works batch-size source; fallback limit detection from error/header bodies should remain in place.

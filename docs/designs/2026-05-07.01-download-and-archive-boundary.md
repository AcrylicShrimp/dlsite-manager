# Download And Archive Boundary

## Decision

Use two crates instead of one combined download/decompression crate:

- `dm-download`: owns DLsite download jobs after `dm-api` has produced a `DownloadPlan`.
- `dm-archive`: owns local archive recognition and unpacking behavior.

The download crate may call the archive crate, but archive logic must not depend on DLsite HTTP, Tauri, storage, or account/session state.

## Legacy Behavior Observed

The old implementation kept download and extraction in `src-tauri/src/services/download_service.rs` and file streaming in `src-tauri/src/dlsite/api.rs`.

Important behavior to preserve conceptually:

- Downloaded files lived under `{download_root}/{work_id}`.
- The old downloader wrote multiple files concurrently and tracked aggregate byte progress.
- Interrupted chunks retried by issuing a new ranged request from the number of bytes already written.
- A single downloaded `.zip` was extracted to a temporary directory, then top-level extracted entries were moved into the product directory and the original ZIP was removed.
- A multi-file old-style split archive was detected when the first file ended in `.exe`; the first file was renamed to `.rar`, extracted with `unrar`, then archive parts were removed.
- Old split extraction flattened one wrapper directory when the temporary extraction directory contained exactly one child directory.
- Failed decompression was logged and ignored by the service so the download could still be considered complete.
- The old database stored only `{product_id, path}` for downloaded products.

Behavior to redesign:

- Do not put HTTP, filesystem writes, extraction, DB writes, and Tauri events in one service.
- Do not delete an existing target directory before a replacement download is safely staged.
- Do not use a fixed `__tmp__` directory name that can collide with product contents.
- Do not silently ignore extraction failure unless the caller explicitly chooses a best-effort policy.
- Do not rely on the closed legacy product metadata endpoint for filenames or file sizes.

## Crate Responsibilities

`dm-download` owns:

- turning `dm-api::DownloadPlan` into a filesystem job;
- metadata probing needed for filenames, sizes, and resume support;
- target-path and temporary-path policy;
- range requests, retries, resume, and atomic finalization;
- progress events, cancellation, and structured errors;
- optional handoff to `dm-archive` after all files are present.

`dm-download` must not:

- parse DLsite API HTML or own account/session logic;
- extract archives directly;
- write storage rows;
- emit Tauri events directly.

`dm-archive` owns:

- archive-set recognition;
- extraction staging;
- path traversal protection;
- wrapper-directory flattening policy;
- cleanup policy for source archives and temporary directories;
- deterministic local fixture tests.

`dm-archive` must not:

- make network requests;
- know about DLsite accounts, sessions, or work metadata;
- update storage;
- emit Tauri events directly.

## Initial Code Surface

The initial scaffold intentionally stops before full stream-to-file implementation because v3 filename policy is not pinned yet.

`dm-archive` starts with archive classification:

- one `.zip` file -> `SingleZip`;
- first file `.exe` with any number of following files -> `LegacySplitRar`;
- otherwise -> keep files as downloaded.

ZIP extraction is now implemented for `SingleZip` plans:

- extracts into a generated `.dm-archive-{N}` staging directory;
- rejects archive entries with parent traversal, absolute/prefix components, backslashes, or NUL bytes;
- optionally flattens a single wrapper directory, matching the old user-facing behavior;
- refuses to overwrite existing output files;
- removes the source ZIP by default after successful extraction;
- cleans staging directories after success and after extraction errors.

`dm-download` starts with job request, progress, cancellation, error types, and small metadata helpers for observed v3 stream responses. Actual file writing should be added after the final target-path and resume policy is designed.

Observed v3 stream metadata:

- Final redirected stream URLs include a `/file/{filename}/` path segment.
- Ranged responses include total size in `Content-Range`, for example `bytes 0-0/204264274`.
- The pinned direct and serial-required works resolved to `.zip` filenames.
- The pinned split work resolved to `.part1.exe`, `.part2.rar`, and `.part3.rar` filenames.
- No `Content-Disposition` header was observed in the pinned ranged responses.

## Download Writer Policy

The initial `dm-download` writer handles one file at a time. A higher-level plan runner should compose it across every file in a `dm-api::DownloadPlan`.

Current writer behavior:

- writes into `{target_dir}/.dm-download/{file_name}` first;
- resumes from the existing staging file length;
- reopens the ranged source from the current byte offset after stream failure;
- finalizes with a rename into `{target_dir}/{file_name}`;
- refuses to overwrite an existing finalized target file;
- validates the filename and rejects path separators, NUL bytes, and non-file path components;
- reports per-file progress after successful chunk writes;
- leaves the staging file in place on cancellation so a later run can resume.

`dm-download` exposes `DlsiteDownloadSource` as the adapter from `dm-api::DlsiteClient` and `dm-api::DownloadStreamRequest` into this writer, while unit tests use a scripted local source.

Plan-level orchestration starts in `dm-download` as `download_work_files`:

- verifies the requested work ID matches the `DownloadPlan`;
- targets `{target_root}/{work_id}`;
- probes each planned file with a first-byte ranged request;
- derives filename from the final `/file/{filename}/` URL segment;
- derives total size from `Content-Range` when present;
- calls the staged writer for each planned file in order.
- when `UnpackPolicy::UnpackWhenRecognized` is selected, hands supported downloaded archive sets to `dm-archive` after all files are present.

Current archive handoff behavior:

- `SingleZip` is extracted by `dm-archive`;
- `KeepArchives` is left untouched;
- `LegacySplitRar` is extracted by `dm-archive` through `unrar-ng`;
- archive extraction errors are job-fatal for `UnpackWhenRecognized`, while source archive parts are preserved on failure.

Legacy split-RAR extraction policy:

- uses `unrar-ng = 0.7.6`, selected after checking current crate availability;
- renames the first `.exe` part to a temporary sibling `.rar` path before listing/extraction, matching the legacy behavior needed by old DLsite split archives;
- validates listed RAR entry paths before extraction and rejects absolute, parent-traversal, backslash, NUL, or empty entries;
- restores the first `.exe` part even if listing or extraction fails;
- removes all source archive parts after successful extraction when `remove_sources` is enabled.

Live verification:

- `crates/dm-download/.env.example` documents the optional live test environment.
- `DMSITE_DOWNLOAD_TEST_LIVE=1` enables a guarded live test that downloads one pinned work to a temporary directory.
- `DMSITE_DOWNLOAD_TEST_MAX_TOTAL_BYTES` prevents accidentally downloading a large pinned work.
- The live test may fall back to the local `dm-api` test credentials/work IDs for convenience, but `dm-download` has its own explicit env names.
- `cargo run -p dm-download --example download_work` performs an explicit real download into the repo-local `.dlsite-downloads` directory by default. That directory is gitignored and intended for manual/live artifact inspection only.
- The real-download example accepts `DMSITE_DOWNLOAD_WORK_ID` or a comma/whitespace-separated `DMSITE_DOWNLOAD_WORK_IDS`, uses `DMSITE_DOWNLOAD_MAX_TOTAL_BYTES` as a per-work safety guard, and only unpacks archives when `DMSITE_DOWNLOAD_UNPACK=1`.
- `RJ296365` should not be treated as a reliable live download fixture right now: DLsite reports the account has rights, but the service can refuse the download because a download limit was reached under high demand.
- `RJ257630` was verified as an available direct ZIP case for this account. It resolved to `RJ257630.zip` at 166.86 MiB and unpacked into `.dlsite-downloads/RJ257630` with `DMSITE_DOWNLOAD_UNPACK=1`.

## Open Issues

- Determine whether the final redirected `/file/{filename}/` segment is reliable across more product categories.
- Decide fallback filenames for direct, split, and serial-required works if the final redirected URL ever lacks a filename.
- Validate ZIP filename encoding behavior against real DLsite archives, especially Japanese paths.
- Validate `LegacySplitRar` against a real available split product. The env-gated test exists, but it needs `DMSITE_ARCHIVE_TEST_LEGACY_SPLIT_RAR=1` and `DMSITE_ARCHIVE_TEST_LEGACY_SPLIT_RAR_DIR` pointed at a gitignored downloaded split-RAR fixture.
- Decide whether decompression failure is job-fatal by default or best-effort by user option.

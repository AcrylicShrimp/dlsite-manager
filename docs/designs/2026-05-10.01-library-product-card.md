# Library Product Card

## Design Goal

Redesign the Library product row into a dense product card that presents the work identity, source credits, ownership, and future download actions without mixing unrelated metadata together.

The card should make maker and source credits easy to scan because they are key DLsite purchase-library context.

## Current Data

The current cached product list exposes:

- work ID;
- title;
- maker name;
- work type code;
- age category;
- thumbnail URL;
- owner account labels;
- purchase timestamps.

The richer DLsite v3 work payload is preserved in `works.raw_json`. The observed `raw_json.tags` entries include source-credit classes:

- `voice_by`;
- `illust_by`;
- `scenario_by`;
- `created_by`;
- `music_by`;
- `other_by`.

These should not be treated as generic chips. They are role-specific credits and should be grouped into readable text rows.

Download state, serial availability, local file paths, and row-level sync are not currently exposed to the product list. Card actions may be designed visually, but functional wiring belongs to later storage/job slices.

## Agreed Shape

Each card has a square thumbnail on the left and a vertical content stack on the right.

```text
[thumbnail]  Title with ellipsis                         RJ01005844
             Maker      Patra Suou
             CV         桐谷華, ヒマリ, あじ秋刀魚...
             Illust     cura, Tam-U...
             Scenario   進行豹
             [Voice/ASMR] [All ages] [Other chips...]

             [Main]                                      [Download] [...]
```

## Content Rules

Title row:

- title is large and single-line truncated with ellipsis;
- work ID is fixed-size, monospaced, and copyable;
- title and work ID expose their full text through tooltip text.

Unified metadata block:

- maker is always first and has its own row;
- source credits follow in a deterministic fixed field grid;
- the card always renders all known fields: `Maker`, `CV`, `Illust`, `Scenario`, `Creator`, `Music`, and `Other`;
- missing values render as a muted `-`, not as `N/A`;
- unknown source-credit classes ending in `_by` are folded into `Other` until they are identified and promoted to a regular field;
- the metadata block is visually tight and continuous;
- each metadata row truncates with ellipsis when there is no room;
- each metadata row exposes its full text through tooltip text.

Credit labels:

| Raw class | Display label |
| --- | --- |
| `voice_by` | `CV` |
| `illust_by` | `Illust` |
| `scenario_by` | `Scenario` |
| `created_by` | `Creator` |
| `music_by` | `Music` |
| `other_by` | `Other` |

Chip row:

- chips are for compact classification/status only;
- source-credit names do not become chips;
- initial chips may include product type and age category;
- product type chips should expand known DLsite work type codes into readable labels and tooltip help, e.g. `SLN` renders as `Simulation` and is grouped as `Game`;
- age chips should expose tooltip help for the DLsite age rating;
- later chips may include local state such as downloaded, discontinued, serial-required, or download progress.

Product type color:

- each card has a narrow vertical color belt on the far left;
- the belt color is based only on product type, never age or download state;
- the product type chip uses the same color tone as the belt;
- age and status chips use their own tones.

Product type tones:

| Product family | Tone |
| --- | --- |
| Audio / Voice / ASMR | yellow |
| Video / Anime / Voice Comic | pink |
| Game / RPG / Adventure / Puzzle / Simulation | purple / violet |
| CG / Manga / Comic / Illust / PDF / reading material | green |
| Tool / Utility / Other / Unknown | gray |

Age chip tones:

| Age category | Tone |
| --- | --- |
| All ages | dimmed green |
| R-15 | dimmed yellow |
| R-18 | dimmed red |

The color system should follow the old DLsite semantic direction without copying raw colors exactly. The dark UI needs softened tones that are readable without making the list visually noisy.

Footer row:

- owner account tags are on the left;
- action buttons are on the right.

## Action Model

The desired final actions are:

- `Open Serial`, when serial numbers are available;
- `Download`, when not downloaded;
- `Open`, when downloaded;
- `Downloading N%`, when a download job is active;
- overflow menu.

Overflow menu:

- refresh/update this work only;
- remove local files, when downloaded;
- download archive only, when not downloaded.

Until download state, serial state, and row-level sync exist in storage/job APIs, these actions should either be disabled placeholders or omitted. The card should not imply a working action that the backend cannot perform.

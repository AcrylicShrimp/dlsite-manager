# Audit Log System

## Decision

Operational audit logs are file-backed support artifacts, not SQLite domain data.

The app writes two kinds of log files under Tauri's app log directory:

- `audit-YYYY-MM-DD.jsonl` for user-support audit events;
- `runtime.log.YYYY-MM-DD` for structured `tracing` diagnostics.

SQLite remains the persistent product/account/settings/download state store. Audit logs are intentionally outside the database so a user can send log files without exporting or inspecting app data tables.

## Audit Events

Audit events are append-only JSON lines with a stable support-oriented shape:

- timestamp;
- level;
- operation name;
- outcome: queued, succeeded, failed, or cancelled;
- human summary;
- optional error code and error message;
- structured details.

Sensitive detail keys are redacted recursively before writing. Redacted keys include password, cookie, token, secret, serial, session, and auth.

The first implementation records:

- app startup;
- settings save success/failure;
- account save and enabled-toggle success/failure;
- account sync and work download queue events;
- account sync and work download terminal job outcomes;
- downloaded work open success/failure;
- downloaded work delete success/failure;
- audit log directory open success/failure.

Read/list operations should not normally produce success audit entries because that would make logs noisy. Their failures can be added later if a concrete debugging need appears.

## Tracing

`tracing`, `tracing-subscriber`, and `tracing-appender` are initialized by the Tauri adapter during app setup.

Runtime tracing logs are JSON-formatted daily files under the same app log directory. They are diagnostic logs for engineers and future agents; audit events remain the support-facing operational contract.

Rules:

- never log passwords, cookies, serial numbers, raw authenticated URLs, or session material;
- prefer operation names, work IDs, account IDs, job IDs, and error codes;
- keep the audit file useful to users and support, and use tracing for lower-level runtime context.

## Tauri Surface

The Tauri layer owns the app log directory and exposes thin commands:

- `list_audit_events(limit)` reads recent audit events from JSONL files;
- `get_audit_log_dir()` returns the app log directory path;
- `open_audit_log_dir()` opens the directory through the guarded native opener.

The frontend Activity page shows recent audit events and exposes an Open folder action so users can locate the files.

## Rejected Alternative

Persisting audit events in SQLite is rejected. Audit logs are operational/support material, not core application state, and file-backed logs are easier for users to attach to bug reports.

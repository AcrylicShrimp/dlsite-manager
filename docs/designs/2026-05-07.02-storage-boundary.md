# Storage Boundary

## Decision

Use a dedicated `dm-storage` crate backed by SQLite through SQLx.

`sqlx 0.9.0-alpha.1` is used because it is the newest crate version currently reported by Cargo. Keep the first storage slice conservative until that dependency choice has more mileage in the project.

The first cut was intentionally small:

- open or create a SQLite database;
- run embedded SQLx migrations;
- expose an explicit write transaction wrapper;
- add only settings needed by the first GUI slice.

The first product-list cache slice now adds account and work ownership tables because the unified Products UI needs them. Download status and serial-number persistence remain deferred until those workflows are built.

## Transaction Rule

All write operations must be enclosed within a transaction.

Storage APIs should preserve that rule structurally:

- migrations use SQLx's migration runner, where normal migrations are transactional by default;
- public mutation APIs should either open their own transaction for one logical write or accept a `WriteTransaction`;
- future multi-step write workflows should start one transaction at the service boundary and pass the transaction through repository methods;
- `dm-storage` must not expose its raw SQLx pool or connection as public API, because callers could otherwise bypass the transaction rule.

## Current Surface

`Storage` owns the SQLite pool and exposes:

- `open(path)` for file-backed databases;
- `open_in_memory()` for tests;
- `run_migrations()`;
- `begin_write()`.
- `app_settings()`;
- `save_app_settings()`.
- `accounts()`;
- `save_account()`;
- `set_account_enabled()`;
- `record_account_login()`;
- `commit_account_sync()`;
- `record_sync_failure()`;
- `record_sync_cancellation()`;
- `sync_runs_for_account()`;
- `list_products()`.

`WriteTransaction` still exposes the minimal static SQL execution hook used by tests, but it now also exposes account/cache write methods for multi-step service operations:

- `upsert_account()`;
- `set_account_enabled()`;
- `record_account_login()`;
- `commit_account_sync()`;
- `record_sync_failure()`.
- `record_sync_cancellation()`.

The account sync write is intentionally atomic: one transaction records the sync run, upserts work details, upserts account ownership, marks missing ownership stale, and updates account sync metadata.

The initial `app_settings` table stores:

- `library_root`;
- `download_root`.

Both settings are nullable at the domain level. Missing values are represented by absent key/value rows.

`library_root` is the final managed library location after download and unpacking. `download_root` is a download staging location, not the final library: it holds partial downloads, retry/resume state, and fetched archives before they are moved or unpacked. The GUI labels this as the download staging folder and defaults it to the platform Downloads directory when no saved value exists.

The product-list cache stores:

- `accounts` for local DLsite account/source metadata, including non-secret `credential_ref`;
- `works` for global cached work details keyed by `work_id`;
- `account_works` for account-specific ownership and purchase timestamps;
- `sync_runs` for completed/failed sync attempts and cache reconciliation.

`accounts.credential_ref` points to `dm-credentials`; passwords are not stored in SQLite.

## Tauri Adapter

The current Tauri adapter initializes `dm-storage` in app setup using the platform app data directory and runs migrations before command registration.

Current commands:

- `get_settings`;
- `save_settings`.
- `list_accounts`;
- `save_account`;
- `set_account_enabled`;
- `list_products`;
- `start_account_sync`;
- `list_jobs`;
- `get_job`;
- `cancel_job`;
- `get_job_logs`;
- `clear_finished_jobs`.

These commands validate/normalize string input, call `dm-storage`, `dm-library`, or `dm-jobs`, and translate errors. They do not own SQL or schema details.

## Deferred

- No session persistence yet.
- No password or serial-number storage in SQLite.
- No download status tables yet.
- No account session persistence yet.
- No API/download GUI workflows yet.

When session persistence becomes necessary, decide whether session snapshots belong in the OS credential store or should remain in memory.

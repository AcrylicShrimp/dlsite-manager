# Manual Auto-Update

## Status

Accepted and first implementation in progress.

## Direction

`dlsite-manager` uses the Tauri 2 updater plugin with GitHub Releases as a static update host. The app does not run startup update checks. Users explicitly start the flow from Settings.

## Release Hosting

Stable builds read:

`https://github.com/AcrylicShrimp/dlsite-manager/releases/latest/download/latest.json`

The release workflow creates updater artifacts and uploads `latest.json` alongside the normal installers. Stable releases are still created as drafts first, so update discovery starts only after the release is manually published. Prerelease/RC tags may upload updater metadata, but the committed stable endpoint follows GitHub's latest non-prerelease release.

## Signing

Tauri updater artifacts are signed separately from OS code signing. The updater public key is committed in `src-tauri/tauri.conf.json`; the private key must remain outside the repository and be provided to release builds through GitHub Actions secrets.

Required release secret:

- `TAURI_SIGNING_PRIVATE_KEY`

Optional release secret:

- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`

The first generated maintainer-local key path is `~/.tauri/dlsite-manager.key`. Losing this private key prevents publishing future updates to already-installed updater-capable builds.

Release automation expects `TAURI_SIGNING_PRIVATE_KEY` to contain the private key file contents, not a repository path. For a local signed bundle, load the file content into the environment and set `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` explicitly, even if it is empty.

## UI Behavior

Settings exposes a manual `Check for Updates` button in the About panel. Pressing it checks the configured endpoint, downloads and installs an available update, then relaunches the app. If no update exists, the user sees a toast and no background polling is scheduled.

## Known Constraints

- Builds before this feature cannot auto-update themselves.
- The first updater-capable release can only update to later releases.
- Current release automation publishes Apple Silicon macOS and Windows x64 updater entries.
- Updater signing does not replace Apple notarization or Windows code signing.
- macOS builds use ad-hoc app-bundle signing (`bundle.macOS.signingIdentity = "-"`) so bundle resources are sealed and updater tarballs/DMGs are structurally valid. This still does not satisfy Gatekeeper for normal quarantined downloads; Developer ID signing plus Apple notarization is required for that.

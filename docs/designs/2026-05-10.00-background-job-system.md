# Background Job System

## Design Goal

Create a reusable background job system for long-running app work such as account sync, downloads, extraction, scanning, and future metadata refreshes.

The system should make jobs visible and controllable from the UI without moving domain workflows into Tauri. It should own lifecycle mechanics: dispatch, process-local tracking, progress snapshots, bounded logs, event publication, and cooperative cancellation.

## Current Problem

The current GUI starts account sync through a blocking `sync_account` Tauri command. `dm-library` already owns sync sequencing and exposes progress/cancellation hooks, and `dm-download` owns download progress/cancellation for file writes. If every feature adds its own blocking command, event name, status model, and cancel path, the Tauri layer will become the orchestration layer again.

## Agreed Architecture

Add `dm-jobs` as a reusable Rust crate.

`dm-jobs` owns:

- job identity;
- job kind/title/metadata;
- lifecycle status;
- cooperative cancellation token;
- current phase/progress snapshot;
- bounded per-job logs;
- process-local active and recent-finished job registry;
- generic job events suitable for Tauri forwarding.

`dm-jobs` does not own:

- DLsite API behavior;
- account sync sequencing;
- download or extraction filesystem behavior;
- SQLite schema or durable domain state;
- Tauri commands;
- frontend-specific presentation.

Tauri owns:

- validating command input;
- starting specific jobs by calling domain crates;
- translating `dm-jobs` snapshots/events to command/event DTOs;
- forwarding generic job events to the frontend;
- keeping the app state instance of `JobManager`.

Domain crates own the actual work:

- `dm-library` owns account sync;
- `dm-download` owns download-to-file behavior;
- `dm-archive` owns extraction behavior;
- `dm-storage` owns durable persistence and transactions.

## Core Model

Job statuses:

```text
queued
running
cancelling
succeeded
failed
cancelled
```

`queued`, `running`, and `cancelling` are active states. `succeeded`, `failed`, and `cancelled` are terminal states.

A job snapshot contains:

- `id`;
- `kind`;
- `title`;
- `status`;
- `phase`;
- `progress`;
- `metadata`;
- `output`;
- `error`;
- `created_at`;
- `started_at`;
- `finished_at`;
- `cancellable`.

Job progress is generic:

```text
current: optional integer
total: optional integer
unit: optional string, such as items, bytes, files
```

Workflow-specific meaning stays outside `dm-jobs`. For example, account sync maps `dm-library::SyncProgress::LoadingWorks { work_count }` to phase `loadingWorks` and progress `{ current: null, total: work_count, unit: "items" }`.

## Events

The Tauri adapter should emit one generic frontend event:

```text
dm-job-event
```

The event payload should mirror `dm-jobs::JobEvent`:

```text
sequence
event_kind
job_id
kind
status
phase
progress
message
log
snapshot
```

Events are ordered by a process-local monotonically increasing sequence number. Events are not durable and do not need to survive app restart.

High-volume workflows, especially downloads, should coalesce or throttle progress events before forwarding them to the frontend. Work-download byte progress is forwarded at roughly one-second intervals, while phase changes and terminal events remain immediate. Download byte progress should represent aggregate work-level bytes across all files in the download plan; per-file counters must not be surfaced as the main job percentage. Metadata probing for individual files is internal setup and should not become a visible Activity/row phase between download parts. The first account-sync integration does not need throttling because phase changes are low-volume.

## Logs And Observability

Each job has bounded in-memory logs. Logs are intended for user-visible diagnostics and short-term debugging, not full telemetry.

Rules:

- never log passwords, cookies, serial numbers, or raw authenticated URLs;
- prefer phase-level milestones and concise error summaries;
- keep logs bounded per job to avoid unbounded memory growth;
- use generic logs for UI diagnostics;
- introduce app-wide `tracing` only when there is a concrete need for broader runtime telemetry.

## Cancellation And Abort Semantics

Normal UI "abort" means cooperative cancellation.

Calling cancel:

1. marks the job status as `cancelling`;
2. flips the job cancellation token;
3. emits a cancellation-requested event;
4. waits for the job body to observe cancellation and return.

Hard task abortion is not part of the normal first-cut UI contract. Force-killing async tasks can interrupt filesystem cleanup, archive cleanup, or transaction boundaries. If hard abort is ever added, it should be reserved for app shutdown or clearly documented emergency behavior.

Domain operations should check cancellation at safe boundaries. Existing sync behavior already checks between login/count/purchase/work/commit phases. Downloads already check during file streaming and preserve staging files on cancellation.

## Persistence Policy

The first `dm-jobs` implementation is process-local and in-memory.

Durable facts remain in domain storage:

- account sync history belongs in `sync_runs`;
- future download state belongs in download-status tables;
- library metadata belongs in library/cache tables.

Generic job history should not be treated as durable truth. On app restart, active jobs are gone, but durable domain tables still explain completed/failed syncs and future download state.

## Tauri Command Surface

First command surface:

```text
start_account_sync(request) -> { jobId }
list_jobs() -> JobSnapshot[]
get_job(jobId) -> JobSnapshot
cancel_job(jobId) -> JobSnapshot
get_job_logs(jobId, afterSeq?, limit?) -> JobLogPage
clear_finished_jobs() -> { removedCount }
```

The old blocking `sync_account` command should be removed or deprecated after the frontend has migrated to `start_account_sync`.

Future commands:

```text
start_work_download(request) -> { jobId }
start_sync_all_accounts(request) -> { jobId }
```

## Account Sync Integration

`start_account_sync` should:

1. validate account ID and optional one-shot password;
2. spawn a `dm-jobs` job with kind `accountSync`;
3. create a fresh `dm_api::DlsiteClient` inside the job;
4. call `dm-library::sync_account_with_source`;
5. map sync progress into generic job phase/progress/log events;
6. return immediately with the job ID.

Completion output metadata should include:

- account ID;
- sync run ID;
- purchased count;
- cached work count;
- observed page limit;
- observed concurrency.

Cancellation should use the job cancellation token and should still let `dm-library` record cancelled sync runs in storage.

## Download Integration Later

Downloads use the same job system for the first single-work integration slice.

Current behavior:

- map `dm-download::DownloadProgress` to generic aggregate byte progress;
- store durable download state in `dm-storage`;
- treat decompression and final movement as phases of the same `workDownload` job.

Still deferred:

- bulk download orchestration;
- dedicated job detail drawer/log UI;
- explicit throttling if real downloads produce too many byte-progress events;
- child jobs for extraction, unless the UI later needs separate extraction control.

## Rejected Alternatives

Persisting generic job state in SQLite is rejected for the first cut. It adds schema and recovery complexity before there is a proven need. Durable domain state already belongs in specific domain tables.

One Tauri event per workflow is rejected. It would recreate fragmented UI state handling and make future downloads/scans harder to present consistently.

Putting job logic in the frontend is rejected. The frontend should present job state and send cancel requests, not infer backend lifecycle transitions.

Hard task abortion as the normal cancel mechanism is rejected. Cooperative cancellation better fits transactional storage, resumable downloads, and archive cleanup.

1. Smoke-test account sync through the GUI with a real account and verify progress, completion refresh, failure display, and cancellation.
2. Add job-backed download commands after the download UI/storage status model is designed.

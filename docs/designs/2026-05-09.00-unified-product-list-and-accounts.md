# Unified Product List And Accounts

## Design Goal

Design the first real GUI-facing library model: a product list backed by cached DLsite v3 data, with multiple DLsite accounts managed as ownership sources and presented as one unified library.

The important product-list rule is:

- accounts are not separate libraries;
- the app shows one virtual "all accounts" library by default;
- duplicate ownership is represented as metadata on one product row, not as duplicate product rows.

## Current State

The rewrite already has the lower-level crates needed for this slice:

- `dm-api` owns DLsite v3 login/session, content count, purchases, works lookup, download plans, authenticated byte streams, and dynamic works batching;
- `dm-download` owns stream-to-file download behavior and hands recognized archives to `dm-archive`;
- `dm-archive` owns ZIP and legacy split-RAR extraction;
- `dm-storage` owns SQLite through SQLx, embedded migrations, app settings, and an explicit write transaction boundary;
- the Tauri layer currently exposes only thin settings commands.

Storage intentionally deferred account, work, purchase, and download tables until a GUI feature required them. The unified product list is that feature.

## Agreed Model

Use a virtual super-account view built from persisted accounts plus cached ownership.

`accounts` are local source records. They represent DLsite login identities that can be enabled, disabled, synced, and shown in ownership badges. They should not imply separate product-list pages.

`works` are global product cache rows keyed by DLsite `work_id`. Each work exists once even when multiple accounts own it.

`account_works` is the ownership join between an account and a work. It stores account-specific purchase and sync information.

The default product list query reads from SQLite only. Network I/O happens during explicit account sync jobs, not while rendering or filtering the list.

## First Schema Shape

The exact SQL can be refined during implementation, but the first cut should stay close to this shape.

```text
accounts
- id TEXT PRIMARY KEY
- label TEXT NOT NULL
- login_name TEXT NULL
- credential_ref TEXT NULL
- enabled INTEGER NOT NULL DEFAULT 1
- created_at TEXT NOT NULL
- updated_at TEXT NOT NULL
- last_login_at TEXT NULL
- last_sync_at TEXT NULL

works
- work_id TEXT PRIMARY KEY
- title TEXT NOT NULL
- title_json TEXT NOT NULL
- maker_id TEXT NULL
- maker_name TEXT NULL
- maker_json TEXT NULL
- work_type TEXT NULL
- age_category TEXT NULL
- thumbnail_url TEXT NULL
- registered_at TEXT NULL
- published_at TEXT NULL
- updated_at TEXT NULL
- raw_json TEXT NOT NULL
- last_detail_sync_at TEXT NOT NULL

account_works
- account_id TEXT NOT NULL REFERENCES accounts(id)
- work_id TEXT NOT NULL REFERENCES works(work_id)
- purchased_at TEXT NULL
- first_seen_at TEXT NOT NULL
- last_seen_at TEXT NOT NULL
- last_seen_sync_run_id TEXT NULL
- is_current INTEGER NOT NULL DEFAULT 1
- PRIMARY KEY(account_id, work_id)

sync_runs
- id TEXT PRIMARY KEY
- account_id TEXT NOT NULL REFERENCES accounts(id)
- status TEXT NOT NULL
- started_at TEXT NOT NULL
- completed_at TEXT NULL
- error_code TEXT NULL
- error_message TEXT NULL
```

`works.raw_json` preserves the observed v3 work DTO so the UI/storage schema can evolve without immediately discarding fields. Common list fields are duplicated into columns for search, sort, and display. DLsite v3 `content/works` currently exposes `content_size`, a byte count for the downloadable content; account sync preserves it so bulk download planning can estimate total bytes from the local cache without resolving download URLs.

Serial numbers should not be stored in this first product-list slice. They are account-specific and should be designed with download/detail workflows and sensitivity in mind.

Download status is now stored once the implementation slice reaches download actions. The first durable state records per-work status, staging path, final local path, unpack policy, byte totals, error details, and timestamps. Active byte progress remains process-local job state; durable storage records enough to render downloaded/failed/cancelled/retry state after the job completes.

## Sync Flow

Account sync is a service operation, not a Tauri command body.

1. Tauri validates input and starts or calls a library sync job.
2. The library service uses `dm-api` to authenticate or reuse an in-memory session.
3. The service calls `content/count` to observe `page_limit`.
4. The service fetches purchases from `content/sales`.
5. The service fetches work details from `content/works`, using the dynamic batch-size behavior already in `dm-api`.
6. The service opens one storage write transaction for the successful cache update.
7. The transaction upserts `works`, upserts `account_works`, records the `sync_run`, and updates account sync metadata.
8. Ownership rows for that account not seen in a complete full sync are marked `is_current = 0`, not hard-deleted.

DLsite can return fewer `content/works` detail rows than requested while still returning a successful response. Account sync should not fail the whole job in that case. Missing detail rows are represented as placeholder `works` rows keyed by the purchased work ID so ownership/cache state is preserved, but the normal Library product list hides those placeholders because they render as confusing empty products. Later successful detail syncs can replace placeholders with richer metadata and make the product visible.

The sync operation should expose structured progress and cancellation from the Rust service layer so the Tauri adapter can forward events without owning the workflow logic.

## Product List Query

The main product list returns one row per `work_id`.

Each row should include:

- product identity and display fields from `works`;
- an ordered list of owning enabled account summaries;
- durable download state when present;
- aggregate purchase metadata, such as earliest/latest purchase date when useful;
- cached API download-size metadata when present;
- enough fields for frontend filters without making the frontend understand the storage schema.

Default scope:

- include current ownership from enabled accounts;
- collapse duplicate ownership by `work_id`;
- exclude disabled accounts from the default unified list;
- preserve disabled-account cache and ownership rows for later re-enable or account diagnostics.

Expected query surface:

```text
list_products(search, account_filters, source_filters, type_filters, age_filters, maker_filters, custom_tag_filters, excluded_custom_tag_filters, sort, limit, offset) -> ProductListPage
list_product_filter_facets(same_query_with_facet-specific_filters_ignored) -> ProductFilterFacets
```

The `account_filters` set is optional. The default is all enabled accounts. Filtering to one or more accounts is a view filter over the same unified model, not a separate account-specific library mode.

The `source_filters` set is optional. Supported source groups are `owned` and `localOnly`. `owned` means the work has current ownership from at least one enabled DLsite account. `localOnly` means the work has no current enabled account owner but has a downloaded local folder/state, so it is present in the local library as not-owned metadata. The frontend presents `localOnly` as a system chip/filter rather than as an editable custom tag.

The `type_filters` set is optional and filters cached `work_type` values into broad Library groups: audio, video, game, image/comic, and other. This grouping should stay storage-backed so later bulk actions can target the same filtered result set that the user sees.

The `age_filters` set is optional and filters cached DLsite `age_category` values. Supported category filters are `all`, `r15`, and `r18`; the unfiltered UI state is separate from DLsite's `all` category.

Maker filters use cached `works.maker_name`. The frontend obtains maker options through a storage-backed facet query rather than deriving them from the current page only. Maker facets should respect search/account/type/age/custom-tag filters while ignoring the currently selected maker filters, so users can switch makers without losing the option list.

Custom tags are user-defined metadata stored separately from DLsite's raw API payload in `work_custom_tags`. Writes replace the full tag set for a work inside one storage transaction, normalize whitespace, deduplicate case-insensitively, and cap individual tag length in storage. Product list rows and product details include the normalized display tags.

Search should cover stable cached columns (`work_id`, title, maker), DLsite tag names preserved in `works.raw_json.tags`, user-defined custom tag names, and source terms such as `local` / `not owned` for local-only works. Custom tag include/exclude filters use normalized names and are part of the same storage-backed query used by Library listing and bulk-download selection.

Bulk download preview should use cached `content_size` from `works.raw_json` for expected-byte totals. It should not call the download endpoint or open authenticated ranged streams for every product just to build the confirmation dialog. If an older cached row lacks `content_size`, the preview may report that row as unknown size until the account is synced again.

Product details should be a projection over the existing cache, not a second persisted detail model. `get_product_detail(work_id)` assembles indexed `works` columns, parsed title/maker variants, parsed credits/tags from `works.raw_json`, custom tags, owners, dates, content size, and durable download state. The UI may present that richer projection in a modal/panel, but should not expose raw cached-field/debug inventories or duplicate the source payload into another table. Custom tag edits are the exception: the detail panel may call a narrow update command that replaces only the app-owned `work_custom_tags` rows for that work.

Local folder scan/import should use `dm-library`'s deterministic work-ID detector before any filesystem import writes are attempted. The detector recognizes known DLsite work prefixes (`RJ`, `VJ`, `BJ`) followed by 6-8 digits, requires ASCII-alphanumeric boundaries, normalizes lowercase prefixes to uppercase, deduplicates repeated IDs in first-seen order, and treats multiple IDs in one folder name as ambiguous for single-work matching.

Local-only import scans direct child directories of the configured library root, imports folders with exactly one detected work ID, and records those folders as downloaded with `unpack_policy = manual`. The scan first creates a safe placeholder from the folder name, then uses DLsite's public `product.json` endpoint to hydrate local/not-owned metadata when available. Hydration may update newly imported rows and existing local placeholders, including earlier offline imports, with title, maker, type, age category, main image, size, dates, creators, and genre tags. This public metadata path must not be treated as proof of ownership or download permission; Play v3 account APIs remain the source for purchases and authenticated downloads. If public metadata lookup fails or returns no product, the placeholder local row remains usable. Existing download records are not overwritten by scan/import; explicit manual mark or re-download actions remain the intentional override paths.

Default Library queries include downloaded local-only rows alongside enabled account-owned rows. Account-specific filters remain account-owned views and exclude synthetic local-only rows, so selecting a DLsite account does not show products that are only present on disk.

## Frontend Shape

The first serious UI should make Library the primary screen.

Recommended layout:

- left navigation with Library, Downloads, Accounts, and Settings as the main group, with Activity pinned at the bottom;
- Library as a focused product-list screen with search, foldable filter groups, sort, and page-local action buttons separated from search/filter controls;
- Downloads as a focused live queue screen for currently queued/running/cancelling download jobs only;
- product list/table with thumbnail, title, maker, work ID, purchase date, and account ownership badges;
- Accounts as the dedicated source-management screen;
- Activity as the dedicated background-job/status screen.

The default visual theme is dark, with neutral charcoal surfaces and restrained accent colors so the app feels like a desktop library tool rather than a marketing page. A future Tailwind migration is acceptable, but the current implementation remains CSS-only until that migration is taken as a separate slice.

Page-level controls should live inside the page surface they affect, not in a global header action area. The workspace header identifies the current view; Library, Downloads, Accounts, Activity, and Settings each own their own reload, sync, clear, or creation controls.

Library scrolling is page-level, not product-table-level. The Library workspace scrolls as one page, while the compact search/action control shell stays sticky at the top of that scroll area. Expanded filters render as normal page content below the sticky shell, so very large maker facet sets remain reachable through the page scroll instead of requiring a viewport-height inner scroller. The product list itself should not introduce a second scrollbar.

The Downloads page is not a durable history page. It filters the process-local job snapshots down to active download jobs (`workDownload` and `bulkWorkDownload`) and hides completed, failed, cancelled, and bulk-preview/planning jobs. Completed/failure diagnostics remain in Activity and audit logs.

Download completion and delete-download actions should not trigger a full Library product-list reload. Those operations change only durable download state for known work IDs, so the frontend patches the affected rows from the command/job result in place. Full list reloads remain appropriate for explicit Reload/search/filter/sort changes and account sync completion, where the visible result set or ownership cache can actually change.

Account management is source management:

- add account;
- edit local label;
- enable or disable account;
- sign in for the current app session;
- sync one account;
- sync all enabled accounts;
- show last sync status and failures per account.

The Accounts screen should prioritize account-source operations rather than mimic a generic settings form. The source list is primary, with page-local reload/sync-all controls, aggregate counts, and compact per-account rows. Each row places the enabled/disabled toggle badge on its own first line, then shows identity on the left, sync/credential/last-sync metadata as a right-side label/value stack, and row actions under that metadata stack. The enabled/disabled state is separate from sync status. Row actions include sync/cancel, update credential, and remove. The account editor is a secondary detail panel used for adding a new source or editing the selected one.

Account removal deletes the account source, its saved app-local credential, account ownership rows, and sync-run history. It does not delete global `works` cache rows, `work_downloads`, downloaded local folders, or staging files. Products that were only owned by the removed account disappear from the default Library unless they have downloaded state; downloaded orphan works remain visible through the synthetic `Local` owner. Remove is blocked in the UI while an account sync job for that account is active.

The UI should not require the user to switch accounts to see products. Account selection should only narrow or explain ownership.

## Session And Credential Policy

Remembered DLsite account passwords are allowed, but they must not be stored in SQLite and they must not use platform keychains for the normal GUI path.

Remembered passwords should be stored through `dm-credentials` in an app-local credential vault under the Tauri app data directory. SQLite may store only a non-secret `credential_ref` that points to a vault entry for the account.

The vault is OS-free by design: the same Rust implementation is used on macOS, Windows, and Linux. It avoids macOS login-keychain prompts, signing/entitlement requirements, and desktop-environment-specific keychain behavior. The tradeoff is explicit: without a user-provided master password, the vault relies on app-directory/file permissions rather than strong encryption.

The GUI does not expose a "remember password" option. Account saves always target the app-local vault; if a saved password is unavailable, the user must update the credential before account sync/download can proceed without a one-shot password.

The account editor also does not expose an enabled/disabled option. New accounts are enabled by default. Existing accounts keep their current enabled state when edited; enabling and disabling remains a row-level source-list action so credential edits do not accidentally change source availability.

`dm-credentials` provides:

- a `CredentialStore` trait for password save/load/delete;
- a production app-local JSON vault implementation;
- an in-memory implementation for deterministic tests.

Authenticated DLsite sessions can remain in memory for the first account/sync implementation. Persisted session snapshots may be considered later, but plaintext session cookies or passwords should not be stored in SQLite.

The Tauri app initializes the local `dm-credentials` vault from the app data directory. Existing account `credential_ref` values remain stable, but users may need to update credentials once when moving from the old keychain-backed experiment to the app-local vault because the secret bytes live in a different store.

Account DTOs should report a saved credential only when `dm-credentials` can actually load the referenced secret from the current vault. A stale SQLite `credential_ref` alone is not enough to show `Credential: Saved`.

## Crate Boundary

Use `dm-credentials` for remembered passwords and `dm-library` for account/product-list orchestration.

`dm-credentials` should own:

- credential references;
- app-local vault file access;
- in-memory credential storage for tests;
- missing-credential behavior, exposed as `Ok(None)`;
- idempotent credential deletion.

`dm-library` owns:

- account-source domain operations;
- product sync orchestration over `dm-api` and `dm-storage`;
- credential lookup through `dm-credentials`;
- product-list query DTOs;
- progress and cancellation types for sync jobs;
- account/session state abstractions that Tauri can host.

`dm-storage` should own:

- migrations;
- repository methods;
- transactional write APIs;
- SQLite query behavior.

Tauri should own:

- input validation and DTO translation;
- job lifecycle registration;
- event emission;
- command exposure.

Tauri should not own:

- DLsite API sequencing;
- batching decisions;
- cache reconciliation;
- SQL details;
- account ownership merge logic.

## Rejected Alternatives

Separate per-account libraries are rejected for the main UI. They force the user to mentally merge duplicate products and conflict with the desired "single giant/super account" behavior.

Fetching the product list live from DLsite on screen load is rejected. It would make the UI slow, brittle, and dependent on login/network state. Sync should update a local cache; list rendering should read the cache.

Storing every v3 field only as raw JSON is rejected for the product list. It would make search, sort, and simple UI queries too awkward. Keep raw JSON for preservation, but duplicate important display/query fields into columns.

Putting sync logic directly into Tauri commands is rejected because it recreates the old maintenance problem: network calls, storage writes, job progress, and UI events become tangled.

## Open Issues

DLsite account identity is not yet proven as a stable API value. Until a stable remote account ID is observed, accounts should use local generated IDs and user-visible labels, with optional login-name storage only as a convenience.

Incremental `content/sales?last=N` semantics are not fully pinned. The first cache implementation should use full account sync for correctness and defer incremental sync until observed behavior is tested.

The exact title fallback order should be chosen during implementation. A likely order is current UI locale, English, Japanese, then first available localized title.

Session persistence remains a later decision. Password storage is separate from session persistence.

Optional stronger vault protection can be considered later with a user-provided master password. Do not silently add platform keychain fallback because it reintroduces OS prompts and cross-platform differences.
